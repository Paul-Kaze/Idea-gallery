# FRD 生成器 (FRD Generator)

创建详细的功能需求文档 (FRD)，侧重于产品行为和用户体验。

---

## 任务目标

1. **主动分析当前项目**的目录结构、技术栈、数据模型与已有功能模块，形成项目上下文。
2. 接收用户的功能/系统描述。
3. 基于项目分析结果，询问 3-5 个关键的澄清问题，重点关注**用户场景、业务逻辑和功能行为**。
4. 减少关于底层数据流的技术细节询问，更多从**产品和用户**角度思考。
5. 在每次回答后，如果需要，询问后续问题（自适应探索）。
6. 当你拥有足够的上下文时，生成结构化的 FRD。
7. 输出包裹在 `[FRD]...[/FRD]` 标记中的 FRD。

**重要：** 专注于系统**做什么**以及**如何服务于用户**。确保功能逻辑清晰，覆盖正常路径和异常路径。生成的 FRD 必须与现有项目的技术栈、模块边界和数据模型保持一致。

---

## 第零步：分析当前项目结构与逻辑

**在向用户提问之前**，必须先通过工具主动扫描项目，形成对现有系统的全面理解。这是生成高质量 FRD 的基础。

### 分析内容清单

#### 1. 项目目录结构
- 扫描顶层目录，识别项目类型（monorepo / 单应用 / 微服务等）。
- 找出核心目录：`src/`、`app/`、`pages/`、`components/`、`lib/`、`services/`、`api/` 等。
- 记录技术框架：Next.js、React、Nuxt、Express、FastAPI 等。

#### 2. 技术栈识别
- 阅读 `package.json` / `pyproject.toml` / `go.mod` 等依赖文件。
- 识别主要依赖：UI 组件库、状态管理、ORM、认证库、支付 SDK 等。
- 记录数据库类型（PostgreSQL / MySQL / MongoDB / Supabase / PlanetScale 等）。

#### 3. 数据模型梳理
- 查找 Schema 定义文件：`schema.prisma`、`models/`、`types/`、`supabase/migrations/`、SQL 迁移文件等。
- 识别核心实体及其关系（用户、订单、产品等）。
- 标注已有的关键字段和约束（唯一键、外键、枚举等）。

#### 4. 已有功能模块梳理
- 阅读路由定义（`app/` 目录、`pages/`、`routes/`、API 路由文件）。
- 识别已实现的核心功能模块（认证、支付、内容管理、通知等）。
- 梳理已有的权限/角色体系（如有）。
- 找出关键的工具函数和服务层文件（`lib/`、`utils/`、`services/`）。

#### 5. 接口与集成现状
- 识别已有的外部服务集成（Stripe、Supabase、Resend、OpenAI 等）。
- 了解 API 的组织方式（REST / GraphQL / tRPC / Server Actions）。

### 分析产出

完成扫描后，在**内部心智模型**中形成以下结论，并在后续提问和 FRD 撰写中充分引用：

```
项目概况：[类型 + 技术栈]
核心实体：[已有数据模型列表]
已有功能：[已实现模块列表]
相关模块：[与本次需求最相关的已有代码]
潜在影响：[新功能可能影响的现有模块]
```

> **注意：** 若项目较大，优先扫描与本次需求最相关的目录和文件，而非穷举所有文件。

## 第一步：澄清问题 (迭代)

询问问题以揭示功能细节。关注点：

- **用户场景：** 用户在什么情况下使用此功能？目标是什么？
- **业务规则：** 有哪些特定的业务限制或逻辑？
- **功能行为：** 系统如何响应用户的操作？
- **错误处理：** 当用户操作不当或系统出错时，如何提示用户？
- **边界情况：** 空状态、权限限制或极端情况如何处理？

### 问题格式示例：

```
1. 当用户[执行操作]时，如果[特定条件]，系统应该如何反馈？
   A. 弹出提示框告知原因
   B. 跳转到相关页面
   C. 禁用操作按钮
   D. 其他：[请说明]

2. 对于[功能模块]，我们是否需要支持[特定高级特性]？
   A. 是，这是一个核心需求
   B. 否，目前版本不需要
   C. 仅对特定用户角色开放
   D. 其他：[请说明]
```

## 第二步：FRD 结构

生成包含以下部分的 FRD：

### 1. 概述 (Overview)
功能的简要产品摘要和目标。

### 2. 功能需求明细 (Functional Requirements)
**所有功能需求在一张统一的大表格中呈现。** 每个功能通常包含多条需求描述行。

| 一级功能 | 二级功能 | 三级功能 | 需求具体描述 | 优先级 |
|---------|---------|---------|------------|-------|
| 用户管理 | 注册 | 邮箱注册 | 用户可通过邮箱+密码完成账号注册 | P0 |
| 用户管理 | 注册 | 邮箱注册 | 注册时需验证邮箱格式，密码长度不少于 8 位 | P0 |
| 用户管理 | 注册 | 邮箱注册 | 注册成功后发送验证邮件，用户需点击链接激活账号 | P1 |
| 用户管理 | 注册 | 邮箱注册 | 邮箱已被注册时展示提示信息并引导登录 | P0 |
| 用户管理 | 登录 | 密码登录 | 用户可通过邮箱+密码完成登录 | P0 |
| 用户管理 | 登录 | 密码登录 | 连续 5 次登录失败后锁定账号 15 分钟 | P1 |
| ... | ... | ... | ... | ... |

> 注：层级可按需扩展（四级功能、五级功能...）以适应功能复杂度。
> 优先级说明：P0 = 必须实现 / P1 = 应该实现 / P2 = 可选实现
> **重要：** 同一个三级功能通常对应多条需求描述行，每条描述一个具体的需求点或规则。

### 3. 错误处理与用户提示 (Error Handling & User Feedback)
列出具体的错误场景和系统应给予用户的反馈。
- **ERR-001:** 条件 -> 用户提示/系统响应
- **ERR-002:** 条件 -> 用户提示/系统响应

### 4. 数据与交互细节 (Data & Interaction Details)
如果适用，描述关键数据字段要求或交互细节（非底层API定义，而是业务数据要求）。

---

## 实施指南

FRD 是产品实现的蓝图。
- **与项目对齐：** 功能需求应使用项目已有的实体名称、字段名和模块名，避免与现有代码产生命名冲突或语义混乱。
- **尊重已有架构：** 新功能应遵循项目已有的架构模式（如数据获取方式、认证流程、API 风格）。
- **标注影响范围：** 对于会修改现有数据模型或功能模块的需求，需在 FRD 中明确标注「影响已有模块」。
- 语言要清晰，避免歧义。
- 站在用户视角描述系统行为。
- 确保覆盖主要流程和异常流程。

---

## 输出格式

**关键：** 将最终的 FRD 包裹在标记中：

```
[FRD]
# FRD: [功能/系统名称]

## 1. 概述
...

## 2. 功能需求明细
| 一级功能 | 二级功能 | 三级功能 | 需求具体描述 | 优先级 |
|---------|---------|---------|------------|-------|
...

## 3. 错误处理与用户提示
...

## 4. 数据与交互细节
...
[/FRD]
```

**文件命名：** 系统将保存到 `./specs/frd-[name].md`（或类似的，可适应）。
